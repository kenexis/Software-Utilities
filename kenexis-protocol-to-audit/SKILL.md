# Kenexis Protocol-to-Audit Builder

## Purpose

Audit protocols are easiest to author and review in markdown — collaborators can comment, version-control plays nicely, and the source-of-truth is human-readable. But the Open-Audit desktop tool requires its own `.aud` JSON format. This skill bridges the gap: it parses a protocol markdown file and constructs a populated `.aud` file using the `kenexis-open-audit-writer` skill, so the same protocol can be authored once and used to seed any number of audit studies.

The skill produces a `.aud` file whose categories, elements, and questions match the markdown's structure, with metadata (assessor guidance, weight, eval type) preserved on each question and ready for an assessor to fill in scores and findings.

## When to use this skill

Trigger on any of:

- The user has a markdown file containing an audit protocol and asks to create a new `.aud` file from it.
- The user mentions "build an audit from this protocol", "scaffold a new audit study", "turn the HAZOP catalog into an Open-Audit file", or similar.
- A workflow requires a fresh `.aud` file pre-populated with a known protocol's questions (e.g., starting an audit on a new client engagement).

If the user wants to **read** an existing `.aud` file, use `kenexis-open-audit-reader`. If they want to **modify** an existing `.aud` file, use `kenexis-open-audit-writer` directly.

## Expected markdown structure

The skill expects a protocol markdown file shaped like this:

```markdown
# Some Protocol Title (optional H1)

…optional preamble paragraphs…

## Module 1 — Module Name

…optional descriptive paragraph…

| ID | Question | Eval | Weight | Assessor Guidance |
|---|---|---|---|---|
| TEAM-01 | Is the chairperson named and credentialed? | hybrid | 5 | Locate the team roster… |
| TEAM-02 | … | programmatic | 4 | … |

…optional pass-criterion line…

## Module 2 — Another Module

| ID | Question | Eval | Weight | Assessor Guidance |
|---|---|---|---|---|
| …|…|…|…|… |
```

Rules the parser applies:

- **Every H2 (`## …`) with a markdown table directly beneath it becomes a Category.** Other H2s (preamble sections like "Conventions" or trailers like "Scoring & Final Disposition") are ignored.
- **The H2 text becomes the Category_Description**, with a leading `Module N — ` stripped if present (so `## Module 1 — Team Composition` becomes the category `Team Composition`).
- **If a module has H3 (`### …`) sub-sections each containing their own table**, each H3 becomes an Element within the Category. Otherwise, the whole module is wrapped in a single Element named `Audit Questions`.
- **Each table row becomes a Question.** The required columns are `ID`, `Question`, `Weight`, and `Assessor Guidance`. The `Eval` column (programmatic / judgment / hybrid) is optional and is preserved as a tag at the start of the Assessor_Notes.
- **The protocol's per-question ID** (e.g., `TEAM-01`) is kept as a `[TEAM-01]` prefix on the Question_Description so it stays visible in the desktop tool. The internal Open-Audit `ID` is a fresh 22-char base36 token, generated by the writer.
- **Weight values 1–5 map to Open-Audit weighting factors** as: 5 → Critical, 4 → Important, 3 → Helpful, 1–2 → Optional. The mapping is configurable via a `--weight-map` argument if your protocol uses a different scale.

The HAZOP protocol catalog (`hazop-audit-protocol-questions.md`) is the reference example for this shape.

## High-level workflow

1. **Confirm the inputs.** The user must supply:
   - A path to the protocol markdown file.
   - A path for the output `.aud` file.
   - Optional study metadata (study name, facility, coordinator, project number, etc.) — defaulted from the markdown's H1 and filename when not provided.
2. **Run the builder script** (`build_audit.py`, included with this skill).
3. **Validate** the resulting file using the writer's validator.
4. **Present the file** to the user with a `computer://` link plus a one-line count summary (categories / questions / weighting distribution).

## Step 1 — Confirm the inputs

If the user named a protocol markdown file but did not specify the output path, default to the markdown filename with `.aud` substituted, written into the workspace folder.

**Study metadata defaults — read this carefully.**

The Open-Audit Overview page contains fields the assessor will fill in for their specific engagement: `Study_Name`, `Study_Coordinator`, `Study_Coordinator_Contact_Info`, `Facility`, `Facility_Location`, `Facility_Owner`, `Project_Number`, `Project_Description`, `Unit`, `Report_Number`, `General_Notes`. The skill must follow these rules:

- **Default every metadata field to empty string**, with one exception: `Study_Name` defaults to the protocol's H1 title (or, if no H1 is present, the markdown filename without extension). This is so the file opens with a recognizable identifier instead of being blank.
- **Never** invent, infer, or fabricate metadata values. Do not borrow names from prior conversations, sample files, the assistant's training data, or the test examples in this document. "Refinery North," "Pat Rivera," "Acme Energy," "2300.001," etc. that appear elsewhere in this skill are illustrative only and **must not** be supplied to the builder.
- **Do not prompt the user for metadata** when starting a new audit from a protocol. The assessor fills these in inside the Open-Audit desktop tool, where they have the right reference materials. Asking up-front interrupts the workflow and risks the assistant misremembering what the user said.
- **Only pass `--study-name`, `--facility`, etc. when the user has explicitly typed those values into the chat for this specific audit.** The bar is "did the user just say it" — not "could it plausibly be inferred."

In practice, the most common correct invocation is:

```bash
python3 build_audit.py --protocol PROTOCOL.md --output OUT.aud
```

with no metadata flags. The Study_Name comes from the protocol's H1; everything else opens blank, ready for the assessor to fill in.

## Step 2 — Run the builder

The builder script is `build_audit.py`. The recommended invocation is the minimal one:

```bash
python3 /path/to/kenexis-protocol-to-audit/build_audit.py \
  --protocol /path/to/some-protocol.md \
  --output   /path/to/study.aud
```

Only `--protocol` and `--output` are required. Every other argument is optional and **must be omitted** unless the user has explicitly told you to populate that field for this specific audit. If the user has supplied metadata in chat, only then pass it through:

```bash
# Only after the user has typed the values verbatim:
python3 build_audit.py --protocol P.md --output O.aud \
  --study-name "<exact value the user typed>" \
  --facility   "<exact value the user typed>"
```

Do not include `--coordinator`, `--facility-owner`, `--project-number`, etc. unless the user has stated them for this audit. Defaults are empty strings.

Important options:

- `--single-element-name "Audit Questions"` — name used for the auto-created Element when a module has no H3 sub-sections. Default is `Audit Questions`.
- `--weight-map "5=Critical,4=Important,3=Helpful,2=Optional,1=Optional"` — override the default mapping from numeric weight to weighting-factor code.
- `--ds-rev 6` — datastore revision to write. Default 6 (matches the SIS Functional Safety Assessment sample).
- `--dry-run` — parse the markdown and report what *would* be written, without producing a file.

Programmatic use:

```python
import sys
sys.path.insert(0, '/path/to/kenexis-open-audit-writer')
sys.path.insert(0, '/path/to/kenexis-protocol-to-audit')
from build_audit import parse_protocol, build_audit_data
from write_aud import save

protocol = parse_protocol("/path/to/protocol.md")
data = build_audit_data(
    protocol,
    study_name="HAZOP Quality Audit — Unit 41",
    facility="Refinery North",
    coordinator_name="Pat Rivera",
)
save(data, "/path/to/study.aud")
```

## Step 3 — Validate

The builder runs `validate(data)` automatically before saving. If the validator surfaces errors, the file is not written and the errors are reported to the user. Likely causes:

- A markdown table has fewer than the required columns (missing ID, Question, Weight, or Assessor Guidance).
- A protocol question has no question text.
- A weight value is not in the configured weight-map.

The user fixes the markdown and re-runs.

## Step 4 — Present the file

After a successful save, finish the response with a single `computer://` link to the new `.aud` file and a short summary:

```
[View your new audit file](computer://path/to/study.aud)

Built from `hazop-audit-protocol-questions.md` — 11 modules, 110 questions.
Weight distribution: Critical 18, Important 39, Helpful 41, Optional 12.
```

Avoid restating the entire protocol; the user can open the `.aud` file and see it.

## Mapping reference

How markdown elements translate into the `.aud` file:

| Markdown | `.aud` location | Notes |
|---|---|---|
| H1 | `Overview.Study_Name` (default if `--study-name` omitted) | Stripped of leading "#" markup. |
| H2 (with a table beneath) | New `Categories[]` entry | "Module N —" prefix stripped from the description. |
| H2 (no table) | Ignored | Preamble/trailer sections like "Conventions" or "Scoring." |
| H3 within a module (with a table) | New `Elements[]` entry inside that Category | Otherwise a single auto-element wraps all questions. |
| Table row | New `Questions[]` entry inside the active Element | One row → one question. |
| Question ID column (e.g., `TEAM-01`) | Prefix in `Question_Description` (`[TEAM-01] …`) | Preserves traceability in the desktop tool's display. |
| Question text column | `Question_Description` | HTML-encoded by the writer. |
| Eval column (`programmatic` / `judgment` / `hybrid`) | Prepended to `Assessor_Guidance` as `[Evaluation: judgment] …` | Optional — omitted if column missing. |
| Weight column (1–5) | `Weighting_Factor_ID` (mapped via `--weight-map`) | Default mapping: 5→Critical, 4→Important, 3→Helpful, 1–2→Optional. |
| Assessor Guidance column | `Assessor_Guidance` | HTML-encoded by the writer. |
| `Assessor_Notes` field | **Always empty at scaffold time.** | Reserved for the assessor's working notes during the actual audit. Never populated by this skill. |

Every question is created with:

- `Selected_Score`: empty
- `Findings`: empty
- `Assessor_Notes`: empty (reserved for the assessor)
- `Selected_Weighting_Factor`, `Weighted_Score`: empty
- `Evidence_IDs`, `Assessor_Recommendation_IDs`: only the canonical `"empty"` sentinel

The canonical scoring criteria (Compliant / Partially Compliant / Non-Compliant) and weighting factors (Critical / Important / Helpful / Optional) are seeded at the file level. The auditor fills in scores, notes, evidence, findings, and recommendations later via the desktop tool or the `kenexis-open-audit-writer` skill.

## What NOT to do

- **Never** edit the markdown protocol file from this skill. It is read-only input.
- **Never** invent, infer, or borrow study metadata. Facility, Facility_Owner, Facility_Location, Coordinator, Coordinator email, Project_Number, Project_Description, Unit, Report_Number — all default to empty string and must stay that way unless the user has explicitly supplied a value in this conversation. Examples in this document and any prior tests are illustrative only.
- **Never** prompt the user for metadata as part of running this skill. The Overview fields are filled in later inside the Open-Audit desktop tool by the assessor. Don't make the user re-state context they'll enter there.
- **Never** override the default `Study_Name`. The default is the protocol's H1 title, which is the right starting point. Pass `--study-name` only if the user has stated a different name verbatim.
- **Never** populate `Assessor_Notes` on any question during scaffolding. That field belongs to the assessor as their working space during the audit. The skill leaves it empty; the assessor fills it in later. Evaluation-type tags and any auxiliary protocol metadata land on `Assessor_Guidance` instead, where they help the assessor decide *how* to evaluate the question.
- **Never** map weight 0 to "Critical" or some other unsafe default. If a weight value is outside the configured map, fail loudly.
- **Never** strip whitespace, decode HTML entities, or modify content from the markdown beyond the documented prefix-stripping. The writer applies its own encoding rules; this skill should pass content through verbatim.
- **Never** skip the validator. If `validate(data)` returns errors, surface them and refuse to save.

## Quality bar

The builder is correct if:

1. Given the HAZOP protocol catalog (`hazop-audit-protocol-questions.md`), it produces an `.aud` file that:
   - Has exactly the same number of categories as the markdown has H2-with-table sections.
   - Has exactly the same number of questions as the markdown has data rows across all tables.
   - Loads cleanly via `kenexis-open-audit-reader` and surfaces every question with the correct text, weight code, and assessor guidance.
2. Saving and re-reading the output is round-trip stable (the writer's `verify_round_trip` returns `ok: True`).
3. Re-running the builder with the same inputs produces a file that differs only in the freshly-generated IDs (deterministic content otherwise).
